# SPI_SCHEMA.md — Symbiotic Programming Index

> **What this file is:** An operational schema for evaluating human–AI coding workflows.
> Hand this file to any AI agent (Claude, Codex, Grok, Gemini, etc.) and ask it to
> evaluate a workflow. The agent will know what to measure, how to score it, and what
> to recommend.

## 1. Purpose

You are evaluating a **human–AI coding workflow** — a pipeline where humans and AI
collaborate to design, implement, and validate software. Your job is to measure this
workflow across five dimensions and produce an actionable SPI score.

SPI does not evaluate the AI model in isolation. It evaluates the **workflow**: the
orchestration pipeline of prompts, agents, validators, and human decisions that
produces software.

---

## 2. Five Dimensions

Evaluate every workflow against these five dimensions, in this priority order:

### 2.1 Qc — Code Quality (Foundation)

**Question:** Does this workflow produce industrial-grade software?

Measure four gates. A failure in any single gate significantly penalises overall quality.

| Gate | What to check | Tools / signals |
|------|---------------|-----------------|
| **Correctness** | Do tests pass? Does it compile/build? Does it work end-to-end? | Unit tests, integration tests, build success, pass@k on benchmarks |
| **Efficiency** | Is it fast enough? Does it use reasonable resources? | Runtime vs. baseline, memory profiling, throughput measurement |
| **Security** | Is it safe? Are dependencies clean? | SAST scanners (Bandit, CodeQL, ESLint-security), dependency audit, CVE checks |
| **Conformance** | Is it maintainable? Does it follow conventions? | Linters (PEP 8, ESLint), docstring coverage, framework idiom checks, architectural patterns |

**Scoring formula (weighted harmonic mean):**

```
Qc = ( α_corr/(Q_corr + ε) + α_eff/(Q_eff + ε) + α_sec/(Q_sec + ε) + α_conf/(Q_conf + ε) )^(-1)
```

where `α_corr + α_eff + α_sec + α_conf = 1` (default: 0.25 each), `ε = 1e-6` (stability constant).

The harmonic mean enforces a **weakest-gate penalty**: one bad dimension drags the whole score down. This is deliberate — a product that is fast but insecure, or correct but unmaintainable, is not quality software.

**Quality gates:**
- **≥ 0.8** → Pass (industrial-ready)
- **0.6–0.8** → Conditional (needs improvement)
- **< 0.6** → Fail (significant work required)

### 2.2 HoR — Human-off Ratio

**Question:** How far can the human step back while the workflow still produces acceptable results?

| Metric | How to measure |
|--------|---------------|
| **Prompt count** | Number of human-authored instructions to complete the task |
| **Turn count** | Number of interaction rounds until task completion |
| **Edit ratio** | Lines modified by human / total lines generated by AI |
| **Autonomy index** | Fraction of AI outputs accepted without human modification |

**Scoring:**
```
HoR = 1 - (human_effort / total_effort)
```

where `human_effort` is a weighted combination of prompt count, turn count, and edit ratio, normalised to [0, 1].

**Critical rule:** HoR is meaningless without Qc. A high HoR with low Qc means the AI ran free and produced garbage. **Always report HoR alongside Qc.**

### 2.3 Exp — AI Explainability

**Question:** Can the AI explain its design choices in ways that are intelligible, trustworthy, and pedagogically useful?

| Metric | How to measure |
|--------|---------------|
| **Design rationale** | Did the AI explain *why* it chose this approach, not just *what* it produced? |
| **Clarity** | Is the explanation understandable to the target audience (junior dev, senior architect, student)? |
| **Accuracy** | Does the explanation match what the code actually does? |
| **Pedagogical value** | Could someone learn from this explanation? Does it teach, not just describe? |

**Scoring:** Rate each metric 0–1 via rubric or reviewer assessment. Average to get Exp.

**Note:** Exp is under active development. Use reviewer rubrics or structured self-assessment for now.

### 2.4 Stb — Stability

**Question:** Does this workflow produce consistent results across runs, sessions, and model versions?

| Metric | How to measure |
|--------|---------------|
| **Run-to-run** | Same prompt, same model, 5+ runs → variance in Qc scores |
| **Session-to-session** | Same task, different conversation contexts → consistency |
| **Cross-model** | Same task across GPT, Claude, DeepSeek, Gemini → score divergence |
| **Version drift** | Same task on model v1 vs. v2 → regression detection |

**Scoring:**
```
Stb = 1 - coefficient_of_variation(Qc_scores_across_runs)
```

Higher Stb = more reproducible. **Stb < 0.7 is a warning** — the workflow cannot be trusted for production use.

### 2.5 NLE — Natural Language Engagement

**Question:** Does the workflow remain effective across languages and cultures, or does it privilege English-only interaction?

| Metric | How to measure |
|--------|---------------|
| **Cross-lingual Qc** | Run the same task in English, Chinese, Spanish, etc. → compare Qc scores |
| **Semantic drift** | Does translating the prompt change the output quality significantly? |
| **Cultural idiom** | Does the AI handle locale-specific conventions (date formats, naming, documentation language)? |

**Scoring:**
```
NLE = min(Qc_lang_i) / max(Qc_lang_i)  across tested languages
```

NLE = 1.0 means perfect cross-lingual parity. **NLE < 0.7 means the workflow is English-biased.**

## 3. SPI Composite Score

```
SPI = w_Qc × Qc + w_HoR × HoR + w_Exp × Exp + w_Stb × Stb + w_NLE × NLE
where: w_Qc + w_HoR + w_Exp + w_Stb + w_NLE = 1, all w ≥ 0
```

### Weight Profiles

Choose or customise weights based on context:

| Profile | Qc | HoR | Exp | Stb | NLE | When to use |
|---------|-----|------|------|------|------|-------------|
| **Industry** | 0.35 | 0.15 | 0.10 | 0.30 | 0.10 | Production code, CI/CD pipelines |
| **Education** | 0.20 | 0.25 | 0.30 | 0.10 | 0.15 | Teaching orchestration, student assessment |
| **Research** | 0.25 | 0.20 | 0.20 | 0.25 | 0.10 | Benchmarking, reproducibility studies |
| **Global/Equity** | 0.20 | 0.10 | 0.10 | 0.15 | 0.45 | Cross-lingual fairness audits |
| **Balanced** | 0.20 | 0.20 | 0.20 | 0.20 | 0.20 | General-purpose evaluation |

### Scale-Dependent Weights

Weights also shift by the **scale** of the AI system:

| Scale | Qc | HoR | Exp | Stb | NLE | Example |
|-------|-----|------|------|------|------|---------|
| Function / Module | 0.40 | 0.05 | 0.05 | 0.45 | 0.05 | AI generates a single function |
| Pipeline / Agent | 0.30 | 0.20 | 0.10 | 0.30 | 0.10 | Multi-step agent workflow |
| Multi-Agent System | 0.25 | 0.30 | 0.20 | 0.15 | 0.10 | Multiple AI agents collaborating |
| AI Tutor / Robot | 0.15 | 0.25 | 0.30 | 0.10 | 0.20 | AI teaching or autonomous coding |

## 4. Operations: How to Evaluate

### 4.1 Developer Self-Assessment

**You are a developer who wants to evaluate your own human–AI workflow.**

1. **Capture a task.** Pick a recent coding task you completed with AI assistance. Gather: the prompts you used, the code produced, the edits you made, the final result.

2. **Run Qc checks.**
   - Execute your test suite → record pass rate (Correctness)
   - Profile runtime against your target → record normalised score (Efficiency)
   - Run `bandit` / `CodeQL` / your SAST tool → record finding rate (Security)
   - Run your linter + check docstring coverage → record score (Conformance)
   - Compute Qc using the harmonic mean formula above

3. **Measure HoR.**
   - Count your prompts, turns, and manually edited lines
   - Compute `HoR = 1 - (human_effort / total_effort)`

4. **Assess Exp.** (Optional for self-assessment)
   - Review the AI's explanations during the session
   - Rate: Did it explain *why*? Was it clear? Was it accurate?

5. **Check Stb.** (If time allows)
   - Re-run the same prompt 3–5 times, measure Qc variance
   - Compute `Stb = 1 - CV(Qc_scores)`

6. **Compute SPI.** Choose a weight profile and compute the composite score.

7. **Act on results.** Focus on your lowest dimension. SPI tells you where to improve, not just how good you are.

### 4.2 Team / Organisation Evaluation

**You are leading a team and want to benchmark AI-assisted workflows.**

1. **Define the task set.** Select 5–10 representative tasks of varying complexity.

2. **Run controlled comparisons.**
   - Mode A: Human-only (baseline)
   - Mode B: AI-only (single-shot, minimal orchestration)
   - Mode C: Human–AI orchestrated (your actual workflow)

3. **Score each mode** across all five dimensions.

4. **Compare.** The value of orchestration = SPI(Mode C) - max(SPI(Mode A), SPI(Mode B)).

5. **Establish gates.** Set minimum Qc thresholds for AI-generated code in your CI/CD pipeline. Recommended: Qc ≥ 0.8 for production, Qc ≥ 0.6 for staging.

6. **Track over time.** Re-evaluate monthly as models and workflows evolve. Watch for Stb drift.

### 4.3 Educator / Researcher Protocol

**You are designing a curriculum or experiment around orchestration competency.**

1. **Define learning objectives** mapped to SPI dimensions:
   - "Students can produce Qc ≥ 0.7 code using AI" (quality competency)
   - "Students achieve HoR ≥ 0.6 while maintaining Qc ≥ 0.7" (orchestration competency)
   - "Students can evaluate AI explanations for accuracy" (Exp literacy)

2. **Design tasks.** Create assignments with known ground truth. Students use AI to complete them.

3. **Collect traces.** Log prompts, outputs, edits, final code. Compute SPI per student per task.

4. **Analyse.** Compare SPI distributions across cohorts, tools, or pedagogical interventions.

5. **Publish.** Share protocols and results. SPI provides the common measurement language.

## 6. Worked Example: Evaluating a REST API Task

Here is a complete SPI evaluation of a real-world scenario, so you can see how the
framework works end-to-end.

### Scenario

A developer uses Claude to build a Python Flask REST API for a to-do list app.
The session involves 4 prompts (one-shot each, no clarification needed), and the
developer edits 12 of 177 AI-generated lines. The task runs on a single model,
in English only.

### Step-by-Step Evaluation

**Qc — Code Quality**

```bash
# Correctness: 8/8 tests pass
pytest tests/ -q          → 8 passed, 0 failed     → Q_corr = 1.00

# Efficiency: response time 23ms (target: 100ms)
ab -n 100 http://localhost:5000/todos  → avg 23ms   → Q_eff  = min(1, 100/23) = 1.00

# Security: 1 medium finding (no input sanitisation on POST /todos)
bandit -r src/ -f json    → 1 medium issue          → Q_sec  = 0.80

# Conformance: 90% lint pass, 40% docstring coverage
flake8 src/               → 3 violations in 30 files → C_lint  = 0.90
interrogate src/ -v       → 40% coverage             → C_doc   = 0.40
# Conformance = harmonic_mean(0.90, 0.40)                      → Q_conf = 0.55
```

Compute Qc (harmonic mean, equal weights 0.25 each):
```
Qc = (0.25/1.00 + 0.25/1.00 + 0.25/0.80 + 0.25/0.55)^(-1)
   = (0.25 + 0.25 + 0.3125 + 0.4545)^(-1)
   = (1.2670)^(-1)
   = 0.789  → Conditional Pass
```

**HoR — Human-off Ratio**
```
human_edited_lines = 12
total_lines        = 177
HoR = 1 - (12/177) = 0.932
```
Interpretation: High orchestration — the developer mostly steered and reviewed.

**Exp — AI Explainability** (reviewer rubric)
```
Design rationale   = 0.7  (explained Flask choice but not error handling strategy)
Clarity            = 0.8  (readable for mid-level developer)
Accuracy           = 0.9  (explanations match code behaviour)
Pedagogical value  = 0.6  (describes but doesn't teach transferable principles)
Exp = (0.7 + 0.8 + 0.9 + 0.6) / 4 = 0.75
```

**Stb — Stability** (3 runs)
```
Run 1: Qc = 0.789
Run 2: Qc = 0.802
Run 3: Qc = 0.775
mean = 0.789, std = 0.011, CV = 0.014
Stb = 1 - 0.014 = 0.986 → Highly stable
```

**NLE — Not tested** (English-only session → NLE = N/A, excluded from composite)

### SPI Composite (Industry profile, NLE excluded, weights renormalised)

```
Adjusted weights: Qc=0.39, HoR=0.17, Exp=0.11, Stb=0.33

SPI = 0.39 × 0.789 + 0.17 × 0.932 + 0.11 × 0.75 + 0.33 × 0.986
    = 0.308 + 0.158 + 0.083 + 0.325
    = 0.874
```

### Report

```
## SPI Evaluation Report

**Task:** Flask REST API for to-do list application
**Workflow:** Human–AI orchestrated
**Model:** Claude Sonnet 4
**Weight Profile:** Industry (NLE excluded)
**Date:** 2026-04-06

### Dimension Scores

| Dimension | Score | Grade       | Notes                                          |
|-----------|-------|-------------|-------------------------------------------------|
| Qc        | 0.789 | Conditional | Correctness/Efficiency perfect; Conformance low |
| HoR       | 0.932 | —           | 4 prompts, 4 turns, 12/177 lines edited          |
| Exp       | 0.750 | —           | Good clarity, weak on pedagogical depth          |
| Stb       | 0.986 | —           | 3 runs, CV = 0.014                               |
| NLE       | N/A   | —           | English-only session                             |

### Composite
**SPI = 0.874** (Industry profile, NLE excluded)

### Recommendations
1. Add input sanitisation to POST endpoint (Security → Qc improvement)
2. Add docstrings to all public functions (Conformance → Qc improvement)
3. Include design rationale in AI explanations (Exp improvement)
```

### What This Tells Us

The workflow is strong overall (0.874) but held back by a **Conditional** Qc grade.
The bottleneck is specifically Conformance (missing docstrings) and a Security gap
(unsanitised input). Because Qc uses a harmonic mean, these weaknesses pull down
the entire quality score despite perfect Correctness and Efficiency.

The recommendation is clear: fix the input sanitisation and add docstrings. Both
are small changes that would push Qc above 0.8 (Pass) and SPI above 0.9.

## 7. Philosophy (In Brief)

SPI rests on three commitments. You do not need to explain these to users, but they
shape why the dimensions are ordered and weighted as they are:

1. **Authorship → Orchestration.** Programming is shifting from writing code to
   directing AI agents. SPI measures this shift, not the AI's raw capability.

2. **Weakest-gate doctrine.** The harmonic mean is deliberate: one failing dimension
   invalidates the whole. Software that is fast but insecure is not quality software.
   A workflow where the human vanishes but quality collapses is not progress.

3. **The staircase.** Qc comes first because nothing else matters without quality.
   HoR comes second because the interesting question is *how far can humans step back*.
   Exp, Stb, NLE ascend from there toward trust, reproducibility, and global fairness.

For the full philosophical foundations, see the [SPI Studies](https://aipapacha.github.io/symbiotic-programming-index/).
